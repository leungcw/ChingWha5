# 微信SEO规则

> 微信公众号的搜索引擎优化规则，涵盖标题、摘要、关键词、标签和完读率优化。

---

## 一、标题优化

### 基本规则

| 规则 | 说明 |
|------|------|
| 字数范围 | 20-28字（含标点） |
| 备选数量 | 每篇文章生成3个备选标题 |
| 前置原则 | 核心关键词前置于标题前10字 |
| 避免全标点 | 标题不以标点开头或结尾 |

### 标题模板

| 类型 | 模板 | 示例 |
|------|------|------|
| 数字型 | {数字}个{方法/技巧/真相}，{效果} | "7个晨间习惯，让你的效率翻倍" |
| 悬念型 | {结果/现象}，{原因你猜不到} | "他35岁辞职创业，原因出乎意料" |
| 对比型 | {A}和{B}，差距在{关键} | "月薪1万和10万的人，差距在这一点" |
| 反常识型 | 不是{常识}，而是{真相} | "不是你不努力，是方向错了" |
| 提问型 | {问题}？{答案/关键} | "为什么你总是存不下钱？答案扎心了" |
| 身份型 | 作为{身份}，我想说{观点} | "做了10年HR，我想说这些大实话" |
| 时间型 | {时间}后，{变化/结果} | "坚持早起30天后，我发现了3个真相" |

### 标题评分标准

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 点击欲 | 40% | 是否制造好奇/悬念/冲突 |
| 关键词 | 25% | 核心词是否前置且自然 |
| 可读性 | 20% | 是否通顺、无语病 |
| 差异化 | 15% | 是否与同类文章标题有区分 |

### 3个备选标题生成规则

```yaml
title_generation:
  count: 3
  strategies:
    - strategy: "click_priority"     # 第一备选：优先点击率
      focus: "悬念/冲突/数字"
    - strategy: "keyword_priority"   # 第二备选：优先SEO关键词
      focus: "关键词前置/精准匹配"
    - strategy: "brand_priority"     # 第三备选：优先品牌调性
      focus: "调性匹配/人格一致"
  length_range: [20, 28]
```

## 二、摘要优化

### 基本规则

| 规则 | 说明 |
|------|------|
| 字节限制 | ≤120字节（中文约40字） |
| 内容要求 | 包含核心关键词+核心价值点 |
| 避免内容 | 不与标题完全重复 |
| 格式 | 1-2句完整短句 |

### 摘要模板

| 类型 | 模板 | 示例 |
|------|------|------|
| 结论型 | {核心结论}，{价值点} | "副业不是赚快钱，而是建立第二技能曲线" |
| 悬念型 | {现象/问题}，{答案在文中} | "为什么越忙越穷？答案藏在3个习惯里" |
| 价值型 | {谁能学到什么}，{具体收益} | "产品经理必看：3个被低估的用户洞察方法" |
| 冲突型 | {反常识观点}，{原因} | "早起的鸟儿不一定有虫吃，关键在时机" |

### 摘要SEO要点

- 核心关键词必须在摘要中出现1次
- 摘要中的关键词与标题关键词保持一致
- 避免使用"本文将介绍""这篇文章"等元语言

## 三、关键词前置

### 基本规则

| 规则 | 说明 |
|------|------|
| 前置范围 | 核心关键词必须出现在正文前200字内 |
| 前置数量 | 至少2个核心关键词 |
| 自然嵌入 | 不可生硬堆砌，需与语境自然融合 |
| 首段要求 | 第一段必须包含主关键词 |

### 关键词布局

| 位置 | 关键词要求 | 示例 |
|------|-----------|------|
| 标题 | 主关键词1次 | "AI工具推荐" |
| 摘要 | 主关键词1次+副关键词1次 | "AI工具...效率提升..." |
| 前200字 | 主关键词1次+副关键词1次 | 正文开头段 |
| 正文中段 | 自然出现2-3次 | 论述过程中 |
| 结尾段 | 主关键词1次 | 收束呼应 |

### 关键词提取规则

```yaml
keyword_extraction:
  primary_keyword: 1     # 主关键词：1个
  secondary_keywords: 2-3 # 副关键词：2-3个
  long_tail_keywords: 1-2 # 长尾关键词：1-2个
  placement:
    title: "primary"
    summary: "primary + 1 secondary"
    first_200_chars: "primary + 1 secondary"
    body: "all keywords, 2-3 times each"
    closing: "primary"
```

## 四、5标签规则

### 标签选择规则

| 标签位 | 选择逻辑 | 示例 |
|--------|----------|------|
| 标签1 | 文章核心话题 | #AI工具 |
| 标签2 | 目标受众 | #职场人 |
| 标签3 | 内容类型 | #效率提升 |
| 标签4 | 行业/领域 | #科技 |
| 标签5 | 热点/趋势关联 | #2025趋势 |

### 标签规则

| 规则 | 说明 |
|------|------|
| 数量 | 恰好5个，不多不少 |
| 字数 | 每个标签2-6字 |
| 唯一性 | 5个标签互不重复，无包含关系 |
| 热度 | 优先选择搜索热度高的标签 |
| 精准 | 标签必须与文章内容强相关 |

### 标签SEO权重

```
标签1 > 标签2 > 标签3 > 标签4 > 标签5
```

标签1的搜索权重最高，必须选择最精准、最核心的话题词。

## 五、完读率优化

### 影响完读率的核心因素

| 因素 | 权重 | 优化方向 |
|------|------|----------|
| 内容节奏 | 30% | 信息密度波+段落节奏 |
| 视觉负担 | 25% | 段落长度+排版留白 |
| 阅读动力 | 25% | 悬念设置+金句分布 |
| 情感投入 | 20% | 情绪曲线+共鸣触发 |

### 完读率优化策略

#### 5.1 开头3秒法则

- 前100字必须出现"钩子"（悬念/数据/冲突/反常识）
- 开头不使用设问（"你是否曾经..."）
- 开头1-2段必须为独立段或短段

#### 5.2 阅读疲劳区优化

- 每800字设置1个"节奏锚点"（金句/小标题/数据冲击）
- 长段落不超过150字（纯文字段落）
- 每3段出现1个短段或独立段

#### 5.3 退出点防控

**高危退出点**：
| 位置 | 原因 | 对策 |
|------|------|------|
| 第3-4段 | "又是这种文章" | 在第3段前放置反常识/意外信息 |
| 中间1/3 | 阅读疲劳 | 缩短段落，插入故事/案例 |
| 金句后 | "说完了" | 金句后紧跟深化/新论据，不给退出信号 |
| 结尾前1段 | 知道要结束了 | 结尾前1段保持信息密度，不松懈 |

#### 5.4 完读率自检清单

```yaml
readability_check:
  - hook_in_first_100_chars: true      # 前100字有钩子
  - rhythm_anchor_every_800_chars: true # 每800字节奏锚点
  - no_paragraph_over_150_chars: true   # 无超长段落
  - short_paragraph_every_3: true       # 每3段1个短段
  - surprise_before_paragraph_3: true   # 第3段前有意外
  - ending_no_sag: true                 # 结尾不松懈
```

## 六、SEO完整检查流程

```yaml
seo_check_pipeline:
  step_1_title:
    - length: 20-28字
    - keyword_front: 前10字含主关键词
    - alternatives: 3个备选
    - score: ≥70分

  step_2_summary:
    - length: ≤120字节
    - keyword_included: 主关键词+副关键词
    - no_meta_language: 无"本文将介绍"等

  step_3_keyword_placement:
    - first_200_chars: 主关键词+副关键词
    - body_density: 2-3次/千字
    - closing_keyword: 主关键词1次

  step_4_tags:
    - count: 5
    - relevance: 强相关
    - no_overlap: 无包含关系
    - order: 核心话题 > 受众 > 类型 > 领域 > 趋势

  step_5_readability:
    - hook_check: 通过
    - fatigue_check: 通过
    - exit_point_check: 通过
```
